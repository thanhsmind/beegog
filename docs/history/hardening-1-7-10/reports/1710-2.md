# 1710-2 — Lock stale takeover requires dead owner pid; 1h hard ceiling; long-hold regression test

**Status:** [DONE]

**Outcome:** `tryStaleTakeover` (lib/lock.mjs) no longer steals a lock on mtime
age alone. Past `STALE_MS`(30s) a lock is only a takeover *candidate*; it is
stolen only when `isPidAlive(holder.pid)` says the owner is dead
(`process.kill(pid,0)`: ESRCH/missing/unparsable = dead, EPERM/any other
errno = alive), unless the age has also passed the new absolute
`HARD_STALE_MS=3_600_000` (1h) ceiling, in which case takeover proceeds
regardless of liveness (pid-reuse guard of last resort). No timer heartbeat
added (locked D2). Header contract prose updated. `.bee/bin/lib/lock.mjs`
synced byte-identical to the templates source (same discipline as prior
hardening cells).

RED was captured first: an intermediate pre-fix variant (isPidAlive exported
but `tryStaleTakeover` left age-only) reproduced the exact live-holder-steal
bug — a busy child stole a 45s-backdated but live-owned lock in 49ms instead
of being refused. After restoring the real fix, both suites are green:
`test_store_lock.mjs` (isPidAlive unit checks + scenarios a-f) and
`test_worktree_cli.mjs` (135/135, including 8 new rows proving a real bee
merge holding `worktree-admin` across a 45s slow verify is never stolen by a
concurrent second acquisition attempt).
