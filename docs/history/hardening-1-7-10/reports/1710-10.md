# 1710-10 — cli-session-adopt-bridge

**Status:** [DONE]

**Outcome:** Threaded `root` into every CLI-level `resolveSessionId` call that previously
hard-refused on a null session id, so cell 1710-5's durable single-live-session adoption
fallback (claims.mjs) actually reaches those call sites. `handleCellsClaimNext` (bee.mjs) now
passes `root`, so `bee cells claim-next` no longer refuses a solo native Codex session (real
`.bee/sessions/<id>.json` record, no `CLAUDE_CODE_SESSION_ID`/`BEE_SESSION_ID` env) — it adopts
that session's identity instead. `reserve()` (reservations.mjs) had the same bug one layer
deeper: it called `resolveSessionId({ flag: session })` without `root`, so a solo session's own
live heartbeat record was read as "another" session by `isConcurrentMode` (which excludes
nothing when no id is passed) and refused `SESSION_REQUIRED` every time. Two-or-more fresh live
sessions still refuse at both call sites, unchanged — the fallback only ever adopts when exactly
one candidate exists. `cells claim` needed no change: its own CLI-level resolve call already
had no `root`, but `claimCellFile` (a layer deeper, fixed by 1710-5) already carries the fallback
itself, so the double-resolution was already a no-op pass-through. `state.session.bind` and
`state.start-feature` take an explicit/optional session id and never hard-refuse on null, so
neither needed a `root` fallback.
