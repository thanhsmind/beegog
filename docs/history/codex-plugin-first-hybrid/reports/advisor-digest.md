# Advisor digest — codex-plugin-first-hybrid (high-risk lane)

Advisor: fable (ceiling tier, AO3/AO13). Consulted 2026-07-18. Verbatim digest:

VERDICT: GO-WITH-CONDITIONS. Direction is right and acceptance criteria are strong, but the plan never names plugin_distribution.mjs — the cleanup pass that will silently delete the very hooks hybrid writes — and its D4/fail-closed guarantees rest on mechanisms that don't exist yet. Conditions 1-3 are blocking; ship them into cph-1/cph-2 scope before Gate 3.

RISK 1 (CRITICAL — self-erasing hooks; plan misses it entirely). The plugin-first cleanup strips `.codex/hooks.json`, not just claude entries. `plugin_distribution.mjs:9` `PROJECT_HOOK_FILES = [".claude/settings.json", ".codex/hooks.json"]`; `cleanHookConfig` (L205-233) removes any entry whose command matches `/.bee/bin/hooks/` or `/hooks/bee-` — exactly what `mergeCodexHooks` writes (`codexHookCommand` → `.bee/bin/hooks/<name>`). In install.sh the order is: onboarding apply writes hooks (line 405) → THEN `$DIST_HELPER ... --apply` cleanup (line 411) strips them. Naive implementation yields the exact skills-only end state D2 forbids, reported as SUCCESS. D5's "helper keeps stripping repo-local claude entries" is wrong-by-omission: the same BEE_HOOK_HANDLERS match hits codex.
MITIGATION: teach collectProjectCleanup to exclude `.codex/hooks.json` from the strip set when runtime includes codex under hybrid (or scope PROJECT_HOOK_FILES by runtime). Add plugin_distribution.mjs to cph-1's Files (currently absent). E2E (a) already asserts hooks-present post-full-install — that catches it, but only if the fixture runs the cleanup step, so make the assertion post-cleanup, not post-onboard.

RISK 2 (D4 ordering is unachievable as literally worded). transition_plugin installs the plugin = skills go live at install.sh:389 (step 4), BEFORE onboarding writes hooks at 405 (step 5). "Hooks land before the skills-side completes" is false for the transition; true only vs the cleanup (411). What actually prevents skills-only from a hook THROW is rollback: apply failure → `handle_transition_failure` (406) → `rollback_plugin` removes the plugin. That path works. But it does NOT cover Risk 1 (cleanup-strip is a success, not a failure, so no rollback fires).
MITIGATION: reinterpret/reword D4 as three achievable guarantees — (a) hooks written before the cleanup pass, (b) cleanup must not strip codex hooks under hybrid [Risk 1], (c) a hook-write failure rolls back the just-installed plugin [exists]. Do not try to reorder transition_plugin after onboarding — plugin-first onboarding depends on the plugin being installed (syncSkills skip).

RISK 3 (fail-closed needs a NEW typed-blocked path). The apply path's only typed refusal is `skillSync.blocked` (applyPlan L2667-2688). `case "merge_codex_hooks"` (L2823-2830) has no try/catch — a write failure throws to main()'s catch (L3229) and emits an untyped `{error}`, exit 1. That satisfies install.sh (nonzero→rollback) but NOT D2's "typed refusal naming repo-copy/hybrid". cph-1's test "injected hook-write failure → typed blocked" has no code to pass against.
MITIGATION: wrap the codex-hook apply in a typed blocked result {status, reason naming repo-copy/hybrid retry, forceable:false}, mirroring the skillSync.blocked shape. Testable in test_onboard_bee.mjs by making `.codex` a file.

RISK 4 (the mechanism is a refactor, not a flag-flip). mergeCodexHooks is invoked only inside `if (repoHooks)` AND `if (!repoOwnsHookCatalog)` (L2450/2470). Plugin-first forces repoHooks=false (L3103). So "stop hard-forcing repoHooks=false" is not enough, and flipping repoHooks=true would ALSO re-enable claude repo-local hooks + settings.json merge (L2458-2465) — violating D5. The codex projection must be HOISTED out of the repoHooks umbrella into a runtime-scoped codex-only sub-path gated on (pluginSource && runtime∈{codex,both}), independent of repoHooks.
MITIGATION: new gate, not a widened repoHooks. Keep claude repo-hook block strictly behind repoHooks as today.

RISK 5 (recheck can't see the hole; claude-only must stay clean). Under plugin-first repoHooks=false, `.codex/hooks.json` never enters subsetManaged/managed (L2571-2617), so install.sh's final `up_to_date` verify (434-442) passes EVEN skills-only — the installer's own gate won't catch a missing/stripped hooks.json. Two consequences: (a) E2E must assert `.codex/hooks.json` on disk directly (plan does — keep it, and add doctor `hooks_file_present:ok`); (b) if the worker adds codex-hooks to the managed set, gate that inclusion on runtime∈{codex,both}&&pluginSource so a claude-only install never starts reporting drift or writing `.codex/hooks.json`.

UPGRADE PATH: existing plugin-first codex installs carry NO managed.repo_hooks record and onboarding.json has no runtime field (hasRepoHooksRecorded L2594; schema L2951-2958). So the codex projection must fire from the passed --runtime, never from recorded state. cph-2 passing `--runtime "$RUNTIME"` on the plugin-first branch (405-branch flags) is load-bearing — verify the `--runtime` passthrough flag is actually threaded into the APPLY invocation at install.sh:405, not only the plan/preview at 360. Default `both` means every plugin-first apply writes `.codex/hooks.json` even claude-focused repos — acceptable only once Risk 1 scoping lands.

D1-D5 AMBIGUITY TO RESOLVE: D4 — "the codex hook projection is written before the plugin transition/cleanup finishes ... a skills-only end state becomes structurally unreachable." The transition finishes BEFORE the hook write (install.sh 389 vs 405); only cleanup follows. Propose amending D4 to the three-part guarantee in Risk 2's mitigation, and add an explicit clause to D5: "the distribution cleanup must NOT strip `.codex/hooks.json` bee entries under codex hybrid" — without it, D5's current wording licenses exactly the Risk 1 bug.

All five mitigations are implementable and testable by cph-1/cph-2 workers. Net: proceed, but cph-1 must include plugin_distribution.mjs scoping + a hoisted codex-hook path + a typed-blocked apply result; cph-2's E2E (a) must assert hooks AFTER the cleanup pass.
