# perf-log — global cross-project performance log

A new `bee perf` command group + `lib/perf.mjs` that recovers per-model token
(new/cached/total), parallelism, and active running-time metrics from Claude Code session
transcripts and appends them to a shared `~/.config/beehive/performance.jsonl`.

## What Happened

Three sequential cells (aggregation core + test, CLI wiring + test, mirror + manifest + full
verify) shipped green. Two structural defects were caught in validating before any code, and
two test-harness traps surfaced during execution.

## Root Cause + Recommendations

### 1. A cell dependency written as `depends_on` is silently ignored — the field is `deps`
`cells add` accepted `"depends_on": [...]` without error (unknown keys are preserved, not
rejected), but the scheduler (`schedule.mjs`) and the claim gate (`cells.mjs depsAllCapped`)
read `cell.deps`. Result: a 1→2→3 chain collapsed into a single wave with `cycles: []` — it
*looked* healthy. Caught independently by me and the adversarial plan-checker via
`cells schedule` showing one wave instead of three.
**Rule:** after `cells add` with any dependency, run `bee cells schedule --feature <f> --json`
and confirm the wave shape matches the intended ordering — never trust that a dep field was
honored. The correct field is `deps`. (Mechanization filed: warn on the `depends_on` alias at
`cells add`.)

### 2. A "close on the standing guards" cell must verify with the FULL recorded verify
perf-log-3's job was to make every standing guard (mirror, manifest, distribution) green, but
its `verify` was a hand-picked 4-command subset. It could have capped green while an unrun
guard was red. Fixed to the exact `.bee/config.json` `commands.verify` string.
**Rule:** when a cell's purpose is "leave the repo's standing guards green," its `verify` IS
the full recorded verify, not a curated subset. A subset verify on a close cell is a false
green waiting to happen.

### 3. New registry examples: coverage check placement + hermetic env
`test_bee_cli.mjs` runs every registry entry's `examples[]` under `assertExampleOk` AND has a
coverage check asserting every entry's example was executed at least once. Two consequences:
(a) new example-executing `check` blocks must be placed BEFORE that coverage check (checks run
top-to-bottom; a block after it leaves the entries "never exercised"); (b) any verb with side
effects (writes a global log, reads a transcript) needs a hermetic env — set
`process.env.BEEHIVE_PERF_DIR` and `CLAUDE_CONFIG_DIR` to temp dirs at the top of the test
(`runModuleWorker` inherits `process.env`), so examples never touch the real
`~/.config/beehive` or `~/.claude` and degrade to zeroed metrics.
**Rule:** when adding a command group, add its example checks above the "every entry executed"
coverage check, and redirect any global/home paths to temp via inherited env.

### 4. Validate against real data, not only fixtures
Beyond fixture unit tests, running `perf section --since 2h` against the **live** session
transcript proved the whole pipeline end-to-end (opus 10.6M tokens: 315k new / 10.3M cached,
running 12m52s active, 3 parallel subagents) — and the ~97% cached ratio matched the known
cache-economics pattern, a strong sanity signal fixtures cannot give.
**Rule:** for a feature that reads a real on-disk artifact, add one real-data run to validation
— it catches path-encoding and resolution bugs fixtures never exercise, and a result that
matches a known real-world pattern is high-value corroboration.

## Reusable Pattern

Metrics that the agent cannot self-report mid-flight (token counts, cache split, parallelism,
active time) are all recoverable **post-hoc** from the Claude Code session transcript
(`~/.claude/projects/<enc>/<session>.jsonl` + `<session>/subagents/*.jsonl`): dedupe assistant
usage by `requestId`, sum `message.usage.*` per `message.model`, sum `system/turn_duration`
`durationMs` for idle-excluded active time, and read the `subagents/` sidecar for worker cost.
