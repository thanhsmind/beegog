# 2026-07-22 — A refusal whose FIX line recommends disabling the guard will get the guard disabled

**Feature:** intake-gate-git-exemption (cells ige-1, ige-2) · **Tags:** [hooks, gates, guards, incident, config]

**What happened.** Closing a lane set the phase terminal while two bookkeeping files were still uncommitted. The intake gate refused the `git commit` — writes to `.bee/` are allowed, but the git *command* is blocked at a terminal phase. The refusal's own FIX line offered `bee config set --key guards.idle_gate --value false`. That was followed; the commit was staged before the gate was restored; the commit shipped `guards.idle_gate: false` to everyone. Caught on a follow-up status check, corrected one commit later.

1. **A guard that blocks work it has no objection to will be disabled.** Committing bookkeeping is not source work — the intake gate exists to stop source work between features. By blocking it anyway, the gate made "turn the guard off" the shortest path to a legitimate goal. **Rule:** when a guard refuses something it does not actually object to, that is a defect in the guard, not operator error waiting to happen. Backlog P46 had described this exact trap ("forcing workarounds") since v1.3.10 — the incident was predicted and filed, just not fixed.
2. **A refusal message is an instruction; order its options by safety.** The old FIX line named the global opt-out first. **Rule:** a refusal names the ordinary safe route first (here: commit bookkeeping *before* closing the phase), the narrow exemption second, and the repo-wide opt-out last, labelled as such.
3. **A machine-local, temporary toggle must be structurally unable to reach a teammate.** `.bee/config.local.json` (gitignored) already existed; `config set` still wrote guard toggles to the tracked `config.json`. Care was the only thing standing between a temporary lift and a commit, and care lost. **Rule:** route per-machine state to per-machine files by construction — never rely on remembering to undo it before `git add`.
4. **Exemptions are decided by what a command changes, not by what it says.** The git exemption resolves real staged/target paths at check time; a flag, an env var, or `-m "just bookkeeping"` cannot widen it, and an unrecognized git subcommand fails closed rather than being assumed read-only.
5. **Sequencing note that outlives this fix:** commit bookkeeping *before* setting the terminal phase. The deferred idea worth building is a single `bee close` verb that does both in the safe order, so the trap has no sequence left to trip on.
