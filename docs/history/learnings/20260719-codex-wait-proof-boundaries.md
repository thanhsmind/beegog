# Learning: Prove the Loaded Boundary, the Live Behavior, and Its Generated Outputs

## Learning: Instruction comprehension is not runtime proof

**Category:** failure
**Severity:** standard
**Tags:** [codex, deployment-boundary, native-trace]
**Applicable-when:** Changing agent-orchestration behavior that depends on an always-loaded instruction surface or a collaboration tool.

## What Happened

The first evidence replay loaded both root doctrine and canonical procedure text, so a correct answer did not prove that the live root boundary was sufficient. It also exercised a questionnaire rather than the native collaboration tools. The repair separated a root-only replay from the future-sync replay and recorded real ordinary-gather and two-reviewer traces that crossed an actual empty timeout, performed the required progress interval, handled completion once, recomputed liveness, and stopped at zero agents.

## Root Cause

Three different claims had been collapsed into one: an agent could understand the rule, the currently loaded deployment surface carried the rule, and the native runtime followed the rule under timing pressure. A combined prose replay could establish only the first claim.

## Recommendation

When changing orchestration behavior, prove each boundary independently: isolate the instruction surface the live runtime actually loads, replay the frozen decision pressure against that surface alone, and capture at least one real native-tool trace for timing and state-transition claims.

## Learning: Generated projections and their ledger are causal feature scope

**Category:** failure
**Severity:** standard
**Tags:** [generated-artifacts, release-manifest, scope]
**Applicable-when:** Canonical skill or doctrine changes render runtime-specific projections whose bytes are release-hashed.

## What Happened

The canonical doctrine repair and its focused judge were green, but the full repository verification exposed stale generated plugin projections and release-manifest hashes. The cell had initially treated those outputs as later close work. It had to expand its declared scope, regenerate both plugin trees, update the hash ledger, and rerun the focused judge, diff check, manifest check, and exact full verification before capping.

## Root Cause

The plan bounded hand-authored sources but did not trace the repository's generation and hashing graph from those sources to release artifacts. The generated bytes were therefore discovered by the final guard instead of owned by validation.

## Recommendation

When a declared source feeds generated projections or a release ledger, include those outputs and their regeneration/check commands in the same cell before execution. Treat them as causal artifacts of the change, never as optional close cleanup.

## Learning: Verify the command that will be recorded

**Category:** pattern
**Severity:** standard
**Tags:** [verification, cell-cap, evidence]
**Applicable-when:** A workflow records a runnable command as the proof used for capping or release.

## What Happened

An earlier stored conjunction exited non-zero even though its constituent commands passed when run separately. The repair replaced the ambiguous proof with a literal focused judge and captured the focused judge, diff check, manifest check, and full repository verification as separate invocations with their own exits and outputs.

## Root Cause

The evidence described the health of individual checks but did not prove the exact shell command stored as the cell's verification contract.

## Recommendation

Before accepting recorded verification, execute the exact stored command literally. Run additional artifact and repository checks separately, and never infer that a failing conjunction passed because its pieces were green in other invocations.
